# Подключение кастомных контроллеров — time-reverse

Как подключить физические контроллеры аркадного автомата к игре:
- **руль (энкодер) → время мира** (крутит время вперёд/назад);
- **кнопки → движение персонажа** (влево/вправо + прыжок).

Статус железа: интерфейс ещё не выбран (HID/геймпад vs Arduino/serial — parked-вопрос
из `GAME_SPEC.md`). Эта дока покрывает **оба** пути, потому что **цель у них одна и та
же — два код-шва**. Меняется только «источник» сигнала, не «приёмник».

Все пути ниже — относительно
`components/unity-game/Assets/_Project/Scripts/`.

---

## Главный принцип: железо зовёт C#-швы, а не «привязывает экшены»

Оба входа в игре читаются **напрямую** с `Mouse.current` / `Keyboard.current`
(жёстко, внутри `Update()`). Шаблонный `InputSystem_Actions.inputactions` для
геймплея **не используется**. Поэтому подключить любой контроллер — значит **подать
значение в код-шов**, а не перепривязать Input Actions. Швов два.

### Шов 1 — РУЛЬ → ВРЕМЯ (готов, менять код не нужно)

Файл: **`WorldTimeController.cs`**

- Сейчас источник — колёсико мыши, `Update()`, **строки 43–51**:
  ```csharp
  Mouse mouse = Mouse.current;
  if (mouse == null) return;
  float scroll = mouse.scroll.ReadValue().y;
  if (scroll != 0f) Advance(scroll * secondsPerScrollUnit);
  ```
- **Публичный шов — вызывай его из железа:**
  ```csharp
  public void Advance(float deltaSeconds)   // строка 58
  ```
  Это единственная воронка времени: и геймплей, и автотесты крутят время через
  `Advance`. Руль на каждый щелчок энкодера зовёт:
  ```csharp
  worldTimeController.Advance(notch * secondsPerScrollUnit);
  // notch = +1 (по часовой) или -1 (против); direction = знак
  ```
- **Масштаб**: сериализованное поле `secondsPerScrollUnit` (по умолчанию `0.12`) —
  сколько секунд мира на один щелчок. Дискретная модель «щелчок = шаг времени»
  постоянна (решение Гейта-1: настоящий энкодер тоже дискретный), поэтому энкодер
  ложится на неё один-в-один: **один детент руля = один щелчок = `Advance(±0.12)`**.
  Точную «цену щелчка» подкрутишь этим полем на плейтесте (энкодеры с разным числом
  шагов на оборот — просто разный `secondsPerScrollUnit`).
- **Сброс** (рестарт экрана): `public void ResetTime()` (строка 66) — время в 0.
  Дёргать не обязательно, это делает игра сама.

> Энкодер **инкрементальный** (шлёт «тик влево/тик вправо»), а не абсолютную позицию —
> это идеально ложится на `Advance(delta)`. Если железо шлёт накопленный счётчик,
> подавай в `Advance` **разницу** с прошлым кадром, а не абсолютное значение.

### Шов 2 — КНОПКИ → ДВИЖЕНИЕ (шов надо ДОБАВИТЬ)

Файл: **`PlayerController.cs`**

- Сейчас источник — клавиатура, `Update()`, **строки 39–62**: `a`/`←` → −1, `d`/`→`
  → +1 (в приватное `inputX`); `Space`/`w`/`↑` при `IsGrounded()` → `jumpQueued`.
  Применяется в `FixedUpdate()` (строки 64–74).
- **Публичного шва пока НЕТ** — ввод читается инлайн в приватные поля. Чтобы внешние
  кнопки его кормили, добавь маленький шов (мимикрия под уже существующие
  `SetControlEnabled` / `TeleportTo` / `CarryBy`):

  ```csharp
  // --- предлагаемое добавление в PlayerController ---
  [SerializeField, Tooltip("Ввод идёт от кастомного контроллера, а не с клавиатуры.")]
  private bool useExternalInput = false;

  /// <summary>Ось движения от внешних кнопок: -1 (влево) .. +1 (вправо).</summary>
  public void SetMoveAxis(float x) => externalMoveX = Mathf.Clamp(x, -1f, 1f);

  /// <summary>Прыжок от внешней кнопки (сработает, если персонаж на земле).</summary>
  public void QueueJump() { if (controlEnabled && IsGrounded()) jumpQueued = true; }

  private float externalMoveX;
  ```
  И в `Update()` — когда `useExternalInput`, брать `inputX = externalMoveX` вместо
  чтения `Keyboard.current` (прыжок в этом режиме ставит `QueueJump()` снаружи).
  Клавиатурная ветка остаётся как есть для дефолтного режима.

> Это изменение геймплей-кода → его стоит провести **инкрементом** (как обычную
> playable-фичу: Гейт 1 → владелец → гейт → плейтест), а не правкой на живую. Сейчас
> это документированная заготовка, не применённая к коду.

## Куда это ставится в сцене

`WorldTimeController` и `PlayerController` — компоненты **в каждой сцене экрана**
(`Screen01.unity`, `Screen02.unity`, `Screen03.unity`). Межэкранных переходов пока нет
(каждый экран самостоятелен, `Play` грузит самый новый).

Заведи один MonoBehaviour-«мост железа» (например, `HardwareInput`), положи его на
объект в сцене, дай ему ссылки на `WorldTimeController` и `PlayerController` (перетащить
в инспекторе или найти в `Start()`), и он на каждый сигнал контроллера зовёт швы:
```csharp
worldTime.Advance(notch * worldTime.SecondsPerScrollUnit); // SecondsPerScrollUnit публичен
player.SetMoveAxis(axis);
player.QueueJump();
```
(Пока экраны отдельные — этот мост надо положить в каждую сцену, либо сделать
persistent-объект с `DontDestroyOnLoad`, когда появится система переходов.)

## Источник сигнала — два пути (общий приёмник = швы выше)

### Путь A — Arduino / serial (COM-порт)

Unity Input System **serial не читает**. Нужен свой ридер:
```csharp
using System.IO.Ports;
// SerialInputReader (набросок):
// - открыть SerialPort(portName, baud) в Start()
// - читать НЕ блокируя главный поток: отдельный тред или ReadExisting()/буфер,
//   НЕ ReadLine() в Update() (заблокирует кадр)
// - распарсить свой протокол (напр. строка "W:+1" — тик руля; "M:-1" — влево; "J" — прыжок)
// - из главного потока вызвать швы (Unity API нельзя звать из фонового треда —
//   складывай в очередь, разбирай в Update())
```
Подводные камни:
- `System.IO.Ports` требует правильного **Api Compatibility Level** в Player Settings
  (проверить: `.NET Framework`, если `System.IO.Ports` не резолвится) — это правка
  ProjectSettings, её проводить осознанно.
- **Задержка (latency)** руль→экран — это тот самый parked-вопрос из `GAME_SPEC`;
  меряется только на живом железе (baud rate, размер пакета, частота опроса).
- Имя порта (`COM3` / `/dev/tty.usbmodemXXXX`) — вынести в поле, не зашивать.

### Путь B — HID / геймпад

Если контроллеры видны ОС как стандартное устройство, читаешь его в том же
`HardwareInput` через Input System:
```csharp
using UnityEngine.InputSystem;
// Gamepad.current / Joystick.current / конкретное устройство по InputSystem.devices;
// руль-энкодер, если приходит как ось — БЕРИ ДЕЛЬТУ по кадрам (ось абсолютна,
// а времени нужен инкремент); кнопки → SetMoveAxis / QueueJump.
```
Подводный камень: геймпад-ось **абсолютна**, а `Advance` хочет **приращение** — считай
разницу с прошлым кадром. Если энкодер приходит как импульсы кнопок «+»/«−» — каждый
импульс это один `Advance(±secondsPerScrollUnit)` (ближе к дискретной модели, и лучше).

## Что решить дальше (parked)

- **Интерфейс**: Arduino/serial vs HID — определяет путь A или B (швы одинаковы).
- **Задержка** руль→экран — приемлема ли; ранний железный прототип (из `GAME_SPEC`).
- **Разрешение энкодера** (шагов на оборот) → подбор `secondsPerScrollUnit`, чтобы
  «поймать момент» ощущалось честно (как на плейтестах экранов 2–3).

## Не сломать тесты

Швы `Advance` (и предложенные `SetMoveAxis`/`QueueJump`) — те же точки, через которые
идут автотесты. Пока железо кормит **швы**, а не лезет мимо них, машинный гейт
(`studio-go suite`) остаётся валиден. Клавиатура/мышь как дефолтный ввод сохраняются —
`useExternalInput` только добавляет источник, не убирает старый.
