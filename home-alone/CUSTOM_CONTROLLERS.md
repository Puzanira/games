# Кастомные контроллеры автомата — куда подключать ввод

Карта слоя ввода для подключения самодельных контроллеров (контроллер кота +
контроллер вазы). Составлено 2026-07-15 по состоянию после инкремента
`vase-minigame` (`9662c0a`).

## Главное

Весь игровой ввод читается **напрямую с устройств Unity Input System** в ТРЁХ
статических методах. В сцене `Apartment.unity` ввод НЕ сериализован — ни
Input Action Asset, ни PlayerInput-компонентов нет. Подключение кастомных
контроллеров = правка только этих трёх мест (или их вынос в общий класс, см.
«Рекомендация» ниже).

| # | Что | Файл : метод | Сейчас читает |
|---|-----|--------------|---------------|
| 1 | **Кот — движение** | `Assets/_Project/Scripts/CatMovement.cs` → `ReadMoveInput()` (строки ~74–92) | WASD/стрелки (`Keyboard.current`), фолбэк — левый стик (`Gamepad.current`). Возвращает `Vector2` (нормализуется выше). |
| 2 | **Кот — действие** (запуск пакости/сценки) | `Assets/_Project/Scripts/CatInteractor.cs` → `ActionPressedThisFrame()` (строки ~61–70) | Space/Enter `wasPressedThisFrame`, фолбэк — южная кнопка геймпада. |
| 3 | **Ваза — толчок** | `Assets/_Project/Scripts/VaseMinigameController.cs` → `PushPressedThisFrame()` (строки ~265–274) | То же: Space/Enter, фолбэк — южная кнопка. |

Кнопка «ЕЩЁ РАЗ» на финальной панели — обычная UGUI-кнопка через
`EventSystem` (InputSystemUIInputModule); для автомата её, вероятно, тоже
захочется повесить на физическую кнопку — это отдельная точка (можно из того же
`ActionPressedThisFrame`-пути вызывать `GameController.RestartMatch()` когда
партия окончена, но это уже изменение поведения → через инкремент).

## Семантика, которую нельзя сломать

- **Движение семплируется в `Update`, применяется в `FixedUpdate`**
  (`CatMovement`): состояние устройств Input System надёжно только в
  dynamic-update контексте. Кастомное чтение — тоже в `Update`-пути
  (внутри `ReadMoveInput`), НЕ в `FixedUpdate`.
- **Заморозка управления идёт выше слоя ввода:** во время сценки
  `GameController` отключает `CatMovement.SetControlEnabled(false)` и компонент
  `CatInteractor`; финал замораживает всё. Если подключаться в три метода из
  таблицы — это работает автоматически, ничего дополнительно проверять не надо.
- **Первый кадр сценки глотается** (`beginFrame` в `VaseMinigameController`):
  нажатие, открывшее сценку, не считается первым толчком. Если у вазы будет
  СВОЯ физическая кнопка (не та же, что «действие» кота), этот guard не мешает.
- Везде используется `wasPressedThisFrame` (фронт нажатия), не `isPressed` —
  для кнопок автомата сохранить ту же семантику (толчок = отдельное нажатие,
  автоповтор не нужен).

## Варианты подключения (от дешёвого к дорогому)

1. **Контроллер эмулирует клавиатуру** (Arduino Leonardo/Pro Micro как HID
   keyboard, кнопочные энкодеры Zero-Delay в keyboard-режиме): код менять НЕ
   надо вообще — замапить джойстик кота на WASD/стрелки, кнопки на Space.
2. **Контроллер определяется как геймпад/джойстик** (HID gamepad): частично
   уже работает — левый стик и южная кнопка читаются как фолбэк. Если раскладка
   нестандартная, в Input System дописывается custom device layout, а в трёх
   методах — чтение нужных контролов (`Joystick.current` и т.п.).
3. **Полностью кастомный HID** (свой протокол): custom device layout в Input
   System + правка трёх методов.

## Рекомендация при реальной проводке

Когда дойдёт до железа — вынести все три метода в один статический класс
`ArcadeInput` (`Assets/_Project/Scripts/ArcadeInput.cs`):

```
ArcadeInput.ReadMove()        → Vector2   (кот)
ArcadeInput.ActionPressed()   → bool      (кот: пакость/запуск сценки)
ArcadeInput.PushPressed()     → bool      (ваза: толчок)
```

— тогда все будущие сценки берут кнопки из одного места, а маппинг
клавиатура/геймпад/кастом живёт в одном файле. Это соответствует GAME_SPEC
(«слой ввода абстрагирован, кастомные контроллеры подбираются позже под готовые
сценки»).

## Тесты (не сломать)

PlayMode-тесты симулируют ввод через `InputTestFixture` (виртуальные
`Keyboard`/`Gamepad`): `CatInputPlayTests.cs`, `VaseScenePlayTests.cs` — с
per-frame переинжектом для «зажатых» клавиш. Правило: кастомные устройства
добавляются как ДОПОЛНИТЕЛЬНЫЙ источник, клавиатура/геймпад остаются рабочими
(это и dev-режим из GAME_SPEC, и то, на чём стоят тесты). Любая правка этих
трёх методов гоняется через `studio-go suite` (48 тестов должны остаться
зелёными).

## Процесс

Сама проводка контроллеров в код — изменение управления, т.е. материальная
дизайн-зона → оформляется инкрементом (Gate 1 → owner → скептик → плейтест),
как обычно. Этот файл — карта, чтобы Gate 1 будущего инкремента и работа с
железом начинались не с археологии.
